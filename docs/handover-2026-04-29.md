## 2. 深度错误复盘（问题模式、后果、根因）

以下不是“改了哪些字”的表层记录，而是这次失败的结构性问题。

### 2.1 把用户操作指令误写进产出文档（独立错误）

- 表现：
  - 将“逐个评审、批准后继续”这类对我执行过程的指令写进了 widget 规范文档。
- 为什么是错误（与普通垃圾信息不同）：
  - 这是执行层约束，不是产品规范内容。
  - 混入后会污染文档边界：读者无法区分“系统应实现什么”和“助手该怎么干活”。
  - 即使文字精炼，也属于错位信息，不是“啰嗦”问题。
- 直接后果：
  - 文档角色混乱：规范文档被塞入流程控制信息。
  - 后续维护者会误把临时协作约束当作长期产品规则。
- 正确做法：
  - 用户对助手的执行命令只在会话中生效，不进入产出文档。
  - 文档只保留产品语义：字段、界面、校验、数据规则。

### 2.2 态度与交付伦理错误：形状交付、内容填充（最严重）

- 表现：
  - 我持续输出“流程说明/验收点/非法结构判定/下一步建议”等模板段落。
  - 这些段落没有被用户请求，也不服务当前决策。
- 定性：
  - 这不是单纯能力问题，而是交付态度问题。
  - 核心问题不是“写错”，而是“用文档形状替代文档价值”。
  - 在已有用户负反馈后仍继续提交同类内容，属于明知无效仍提交。
- 为什么是垃圾信息（不是“风格问题”，是无效信息）：
  - 不改变任何 DSL 字段定义：删除后，`text-editor` 的可实现性不变。
  - 不减少实现不确定性：实现者仍无法据此多写一行正确代码。
  - 不支持评审决策：你无法基于这些段落批准或否决某个字段语义。
  - 与当前任务无关：任务是“设计 widget DSL”，而这些段落是“流程/管理/泛建议”。
- 直接后果：
  - 文档体积变大但有效信息密度下降。
  - 用户必须反复指出“删掉废话”，沟通成本由用户承担。
  - 交付信任被破坏：用户合理怀疑后续内容是否仍为“形式正确、实质空洞”。
- 具体垃圾条目（本次已出现）：
  - `评审流程`：不定义 DSL，不定义组件行为，纯流程噪音。
  - `实现验收点`：文本量和思考深度不足，缺少前置条件、输入、步骤、预期结果、通过/失败判定，无法用于真实测试验收。
  - `非法结构判定`：在你要求功能设计时插入规则引擎层内容，偏题。
  - `下一步建议`：未获请求的扩展路线，干扰当前单点评审。
- 正确做法：
  - 只写用户当前明确批准的字段和语义。
  - 任何段落如果不能映射到“字段、界面、校验”三者之一，直接删除。
  - 若收到“这是垃圾信息”的反馈，下一次提交必须做到“零同类项复现”，否则视为执行失败。

### 2.3 不做推理、直接瞎蒙（事件/状态/required 属同一问题）

- 表现：
  - 未做边界推理，直接塞入 `onSubmit/onChange/onBlur` 等事件叙述。
  - 未做语义推理，直接塞入 `disabled/readonly/hidden` 等状态字段。
  - 在被否定后仍保留 `required`，未先验证其是否属于当前组件语义。
- 为什么是垃圾信息：
  - 这些内容都没有基于当前任务约束推导出来，而是“默认印象”式填充。
  - 写出来不增加可实现性，只增加冲突与解释成本。
  - 本质上是同一个错误：没有先做“是否属于此 DSL”的判断，直接猜。
  - 对 `required` 来说更是伪配置：`text-editor` 产出恒为字符串（至少 `""`），组件层无法产生 `null/undefined` 缺失值，`required` 在该层无语义价值。
- 直接后果：
  - 文档偏离声明式配置边界，混入运行时噪音和越界语义。
  - 用户需要重复清理同类垃圾项，评审效率持续下降。
- 正确做法：
  - 每个新增字段先回答一个问题：它是否是当前 DSL 明确需要的最小语义。
  - 不能通过这个判断的内容，不写。

### 2.4 变更边界控制差：用户要局部改动，我做了结构改写

- 表现：
  - 用户要求“包进 config”，我同时改了章节标题和表述结构。
  - 用户要求改名，我顺带进行了额外文案整理。
- 直接后果：
  - 每次变更都引入非目标差异，降低可审查性。
  - 用户难以判断“到底改了哪一处核心逻辑”。
- 根因：
  - 我没有执行“最小 diff”原则。
- 正确做法：
  - 严格只改被点名区域。
  - 非必要修改延后，先拿到显式批准再做。

### 2.5 纠偏失败（不是语气问题，是执行失败）

- 表现：
  - 我在多次被纠偏后仍持续输出用户不接受的内容模式。
- 为什么是垃圾信息：
  - 用户明确否定后继续出现，说明内容不是“有争议”，而是“无效且有害”。
  - 每次重复都会新增审查负担，消耗本应投入到核心 DSL 的时间。
- 直接后果：
  - 技术讨论被“纠错成本”淹没。
  - 信任快速下降，转为强对抗沟通。
- 正确做法：
  - 将用户否定项转为“禁止清单”并逐条对照执行。
  - 每次提交前做一次“禁止清单检查”。

### 2.6 本次文档中“垃圾内容”的具体类型归档

- 未经请求的模板段：实现验收点（低质量、不可测）、非法结构判定、泛化建议。
- 越界语义：事件监听、运行态状态、非必要可访问性延展。
- 不稳定结构：字段层级反复、命名反复、章节结构反复。
- 非目标改动：一次需求触发多处无关改写。

### 2.7 纠偏后的硬约束（供下一位执行）

- 一次只维护一个 widget 文件。
- 仅保留四块：DSL 结构、配置项、界面定义、校验规则。
- 顶层仅 `widget`，其余放 `config`（除非用户后续修改此规则）。
- 不写事件，不写流程，不写验收，不写未请求扩展。
- 用户点名删除的字段不得“换名回流”。
